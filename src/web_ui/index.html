Battery allocation

Multiplex layout: at most one battery per circuit is active at any time — guaranteeing the shared protective device cannot be overloaded. Active leads share the raw CT readings, weighted by each lead's hardware cap in the current direction. Direction is inferred from the active lead's SoC trend (HA polling), not from the CT (which is bent by the battery's own action). Lead switches honour a 30 s settle so the previous inverter shuts off via its CT-watchdog before the new one starts.

Phase detection

Briefly drives each battery alone to detect which CT phase it sits on. Pauses normal dispatch for ~50 s per battery — kick it off when nothing critical is happening on the grid. Result is persisted to config.toml and used by the stuck detector and (later) by smarter routing.

Configuration

Edits are validated, persisted to the TOML file (–) and applied live where possible. Settings tagged restart required are written to disk but only take effect after a service restart. TOML comments are not preserved when the file is rewritten.

Dispatcher (saldierender Passthrough)

The CT readings are forwarded to each battery scaled by its weighted share (sums to 1.0 across eligible batteries). The inverter's own control loop drives its output to compensate. Important: ensure each battery's hardware discharge / charge limit is set to the same value as max_charge_w / max_discharge_w in its config — otherwise the wiring protection enforced here may not match what the inverter will actually deliver.

Home Assistant integration

Optional: read battery SoC from HA entities instead of polling the inverter directly. Useful when an HA integration (e.g. a Marstek HACS plugin) already owns the inverter's UDP port. The URL must be reachable from the add-on container — we always run with host_network: true, so the hassio-internal http://supervisor/core/api alias does not resolve. Use http://homeassistant.local:8123/api (HA-OS mDNS name) or replace with the HA host's IP if mDNS doesn't work for you. Token can be left blank inside the add-on — it's then taken from $SUPERVISOR_TOKEN, which HA Core accepts as a Bearer token directly.

All HA fields take effect only after an add-on restart. The HA poller spawns its tasks at startup based on the current config; flipping `enabled` or changing `soc_entity_id` while the add-on is running has no effect until the next restart.

Circuits

A circuit is a shared protective device (MCB/RCD) downstream of which one or more batteries hang. The dispatcher guarantees at most one battery per circuit is active at a time, so the cap (= phases × fuse_amps × voltage) only needs to be ≥ the largest single member's max(charge_w, discharge_w). Validated at startup — violations refuse the start with a clear error message.
